# @dexterai/vault 0.4.2 — Phase 1 account-list update Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax.

**Goal:** Update the `@dexterai/vault` SDK's `register_session_key` and `finalize_withdrawal` instruction builders to match the new on-chain account lists Phase 1 introduced (live-USDC-balance gating accounts), so the published client stack works against the upgraded program. Stage build-only; publish happens in the combined Phase 1 deploy window.

**Architecture:** Phase 1 added a `vault_usdc_ata` (and for register, also `swig` + `swig_wallet_address`) to two existing instructions so the reservation/overcommit invariants can read the LIVE USDC balance on-chain. The instruction *data* (discriminator + Borsh args) is UNCHANGED for both — only the *account lists* grew. The SDK builders currently emit the old account lists (0.4.1), which the upgraded program will reject. This bumps them to match, plus byte-parity coverage, then bumps to 0.4.2.

**Tech Stack:** TypeScript, `@solana/web3.js`, vitest (byte-parity), the deployed program IDL as source of truth.

---

## AUTHORITATIVE SOURCE — the new account orders (from the generated IDL, verified)
```
register_session_key:                  finalize_withdrawal:
  0. vault (writable)                     0. swig
  1. vault_usdc_ata        ← NEW          1. swig_wallet_address
  2. swig                  ← NEW          2. vault (writable)
  3. swig_wallet_address   ← NEW          3. vault_usdc_ata        ← NEW
  4. instructions_sysvar                  4. instructions_sysvar
```
- `vault_usdc_ata` = the swig wallet's USDC associated token account (the on-chain handler reads `.amount` live and cross-checks `.owner == swig_wallet_address`).
- `swig` = address-constrained to `vault.swig_address`.
- `swig_wallet_address` = PDA `[SWIG_WALLET_ADDRESS_SEED, swig]` under `SWIG_PROGRAM_ID`.
- Instruction DATA is unchanged for both (same discriminator, same Borsh args incl. `max_revolving_capacity` for register).

## CONTEXT — why this exists / what it does NOT change
- This is the SDK consequence of Phase 1's reservation gating. It does NOT touch the on-chain program (already built by the other agent's Phase 1 work).
- The register message bytes (188-byte V2 passkey-signed payload) are UNCHANGED — `sessionRegisterMessage` stays exactly as in 0.4.1. Only the INSTRUCTION ACCOUNTS change, not the signed message.
- `settle_tab_voucher` did NOT gain an account (its frontier guard reads in-account `crystallized_cumulative`), so its SDK builder is unchanged. Verify this assumption holds (Task 0).
